PURPOSE: Core dice engine. Implements the Alternity d20 + situation die system.
         Lower roll = better. Bonus dice subtract, penalty dice add.
         (PHB pp.13, 49: Situation Die Steps Scale and full chart)
DEPENDENCIES:
  Reads: global.config.dice.steps (array of step config structs from config.json)
  Writes: nothing (pure functions, returns result structs)

MACROS:
  SIT_STEP_MIN  = 0   (maximum bonus: -d20)
  SIT_STEP_BASE = 5   (neutral: +d0, no situation die)
  SIT_STEP_MAX  = 13  (maximum penalty: +4d20)

FUNCTIONS:
   1. get_step(index) — line 10 — Returns step config struct {sides, count, bonus, name} clamped to valid range
   2. situation_die_sides(step) — line 17 — Returns die sides for a step (0/4/6/8/12/20)
   3. situation_step_is_bonus(step) — line 18 — Returns true if this step subtracts from d20 (bonus)
   4. situation_step_name(step) — line 22 — Returns display string like "+d8" or "-d4" or "+2d20"
   5. roll_die(sides) — line 25 — Returns irandom_range(1,sides), or 0 if sides<=0
   6. roll_situation(step) — line 28 — Rolls situation die (handles multi-die like +2d20)
   7. alternity_check(score_ord, score_good, score_amz, situation_step) — line 38 — THE core check (PHB p.49). Rolls d20 + situation die, compares to thresholds. Returns result struct with degree -1(crit fail) to 3(amazing), degree_name, total, control/situation rolls
   8. alternity_action_check(ac_ord, ac_good, ac_amz, situation_step) — line 56 — Action check variant (PHB p.38, Table P7): failures become MARGINAL (degree 0) instead of failure

HOW IT CONNECTS:
  Called by meta_roll() and _execute_instant() in scr_ui_helpers. Never called
  directly from UI code. The situation_step integer is the ONLY input that
  controls difficulty — all modifiers (training, FX, wounds) are converted
  to step offsets before reaching this function.

DATA FLOW: How a roll goes from click to result
================================================================================

Example: Player clicks a skill on the Character tab, then clicks ROLL.

1. STAGE (Step_0 line 317-326):
   - btn_clicked("roll") fires
   - No staged_roll exists, so prepare_roll() is called:
     a. build_skill_request(hero, broad, spec, 0)
        - Finds skill index, reads scores, determines training penalty
        - Returns request struct {roll_type, name, score_ord/good/amz, base_penalty, training, broad_skill, spec_skill, ability}
     b. prepare_roll(hero, request):
        - Calls annotate_training() — adds "Broad+1" or "Untrained+1" to modifiers
        - Calls apply_fx_modifiers() — THE modifier engine:
          i.   get_roll_keywords(request) — generates tags: ["str", "skill", "trained", "athletics", "climb", "physical"]
          ii.  Loops hero.fx array — for each active FX:
               - Looks up global.fx_database via get_fx_data()
               - Resolves keywords+modifier based on: keyword_tiers > quality_scale > fixed
               - Checks if ANY FX keyword matches ANY roll tag
               - If match: adds modifier to base_penalty, annotates in modifiers[]
        - Adds wound penalty if enabled
        - Computes final step: clamp(SIT_STEP_BASE + base_penalty)
        - Stores in obj_game.staged_roll, sets situation_step to computed value

2. PLAYER ADJUSTMENT (optional):
   - Player can click </> to adjust situation_step
   - The delta between their adjustment and computed is tracked

3. EXECUTE (Step_0 line 317-326, after animation timer):
   - execute_staged_roll():
     a. Calculates player_delta = current situation_step - computed_step
     b. If delta != 0, annotates "Player Mod+N" in modifiers
     c. Calls alternity_check(score_ord, score_good, score_amz, final_step):
        - Rolls d20 (control die)
        - Rolls situation die (get_step -> roll_situation)
        - total = max(1, control +/- situation)
        - Determines degree: crit fail if d20=20, else compare to score/thresholds
        - Returns {control_roll, situation_roll, total, degree, degree_name, is_critical_failure}
     d. apply_cant_fail() — if cant_fail_mode on, failure becomes MARGINAL
     e. log_roll(name, result):
        - Sets obj_game.last_roll for display
        - Inserts into roll_log array (capped at max_log_entries)
        - Calls append_roll_log_file() — writes to persistent {name}_rolllog.log
        - Sets rolllog_dirty = true
     f. For attacks: track_hit() saves weapon+degree for damage follow-up
     g. Clears staged_roll

4. DISPLAY (Draw_64 right panel):
   - last_roll is rendered: control die, situation die, total, degree (color-coded)
   - Modifiers are listed below result
   - roll_log shows recent history

QUICK ROLLS (right panel buttons like Awareness):
  Skip staging. Call quick_roll_skill(hero, request):
  - Start from player's current situation_step (not base)
  - Apply auto-mods ON TOP of player's setting
  - Roll instantly via _execute_instant()
  This means: player sets the die, auto-mods adjust from there.


================================================================================
FX SYSTEM: How modifiers are matched and applied
================================================================================

The FX system is DATA-DRIVEN. No hardcoded modifier logic. Everything is in
fx_database.json. To add a new modifier, add a JSON entry — no code changes.
Source rules: PHB Table P26 pp.104-107 (Perks), Table P27 pp.108-111 (Flaws),
Table P53 pp.241-245 (Cybertech). Modifiers expressed as situation die step offsets.

HERO.FX ARRAY:
  Each hero has an .fx array. Each entry: { name, type, quality, active }
  Types: "perk", "flaw", "cybertech" (future: "racial", "environmental", "item")
  Quality: "" (flat), "O" (Ordinary), "G" (Good), "A" (Amazing)
  Active: true/false toggle (controlled by radio buttons on Character tab)

GLOBAL.FX_DATABASE:
  Array of full FX definitions loaded from fx_database.json.
  Each entry has: name, type, cost, description, keywords[], modifier, and
  optionally: quality_scale, keyword_tiers, prereqs, size, category.

THREE MODIFIER RESOLUTION MODES (checked in this order):

  1. KEYWORD_TIERS — Per-quality keyword gating:
     Different quality tiers unlock different keyword sets AND modifiers.
     JSON: "keyword_tiers": { "G": { "keywords": ["ranged","combat"], "modifier": -1 },
                              "A": { "keywords": ["ranged","combat","awareness"], "modifier": -2 } }
     At Ordinary quality: no keywords match. At Good: ranged/combat get -1.
     At Amazing: ranged/combat/awareness all get -2.
     Used for: cybertech that improves with quality (e.g., Optic Screen).

  2. QUALITY_SCALE — Same keywords, different modifier per quality:
     JSON: "keywords": ["awareness"], "quality_scale": { "O": -1, "G": -2, "A": -3 }
     Keywords always match. Only the modifier value changes per quality tier.
     Used for: perks/cybertech with graduated effects.

  3. FIXED — Flat keywords + flat modifier:
     JSON: "keywords": ["awareness","perception"], "modifier": -1
     Always matches those keywords. Always applies that modifier.
     Used for: most perks and flaws (e.g., Observant -1 to awareness checks).

KEYWORD TAG GENERATION (get_roll_keywords):
  Every roll request gets tagged automatically:
  - Ability key: "str", "dex", "con", "int", "wil", "per"
  - Roll type: "skill", "feat", "attack", "initiative"
  - Training: "trained", "broad", "untrained", "feat"
  - Broad skill name (lowercased): "awareness", "athletics", etc.
  - Specialty name (lowercased): "perception", "rifle", etc.
  - Category tags from keyword_tree.json: "combat", "ranged", "melee",
    "psionic", "social", "tech", "physical", "vehicle", etc.
  - Attack rolls always get "combat" tag.

MATCHING:
  apply_fx_modifiers() loops ALL active hero.fx entries. For each:
  1. Skip if inactive
  2. Look up full definition from global.fx_database
  3. Resolve keywords + modifier via the 3-mode priority above
  4. Check if ANY resolved keyword matches ANY roll tag
  5. If match and modifier != 0: add to base_penalty, annotate in modifiers[]

EXTENDING:
  To add a new modifier (no code changes):
  1. Add entry to fx_database.json: { name, type, keywords, modifier/quality_scale/keyword_tiers }
  2. Done. apply_fx_modifiers() will auto-match it on every relevant roll.
  Future hooks: environmental modifiers, GM hidden modifiers — just more FX entries.

  .career             string    — Career name (e.g., "Infantry Soldier") (PHB pp.128-130)
  .species             real     — SPECIES enum (0-5) (PHB pp.21-31, Table P3 p.34)
  .secondary_profession real    — PROFESSION enum or -1 (Diplomat dual-class only)
  .portrait_path      string    — Custom portrait file path (relative)
  .background         string    — Free-text background/bio

ABILITIES (6 structs, identical shape):  (PHB p.34, Table P3: species limits)
  .str / .dex / .con / .int_ / .wil / .per
    .score             real     — Ability score (4-14 for humans)
    .untrained         real     — score div 2 (for untrained checks, PHB p.63)
    .res_mod           real     — Resistance modifier: (score div 2) - 5 (PHB p.33, Table P2)

  NOTE: Intelligence is .int_ (not .int) because "int" is a GML keyword.

ACTION CHECK:  (PHB p.38, Table P7: Actions Per Round)
  .action_check
    .marginal          real     — ordinary + 1 (displayed as "X+")
    .ordinary          real     — (DEX + INT) / 2 + profession bonus
    .good              real     — ordinary div 2
    .amazing           real     — ordinary div 4
  .actions_per_round   real     — Usually 2

DURABILITY (3 tracks, identical shape):  (PHB pp.40-41: How to Record Durability)
  .stun / .wound / .mortal
    .max               real     — CON + profession durability mod (mortal = half)
    .current           real     — Current remaining (damage = max - current)

SKILLS (array of skill structs):  (PHB pp.63-66, Tables P18/P19)
  .skills[i]
    .ability           string   — "str"/"dex"/"con"/"int"/"wil"/"per"
    .broad_skill       string   — Broad skill name (e.g., "Modern Ranged Weapons")
    .specialty         string   — Specialty name or "" if broad-only
    .rank              real     — 0-3 (broads always 1, specs 0-3)
    .score_ordinary    real     — Computed: ability + rank adjustments
    .score_good        real     — ordinary div 2
    .score_amazing     real     — ordinary div 4

  SAVE FORMAT: broad = ["Athletics", "STR", -1], spec = ["Rifle", "Modern Ranged Weapons", 2]

WEAPONS (array of weapon structs):  (PHB Tables P38-P40 pp.173-184)
  .weapons[i]
    .name              string   — Weapon name
    .skill_keyword     string   — Maps to keyword_tree.weapon_skills (e.g., "rifle", "pistol", "blade", "brawl")
    .dmg_ordinary      string   — Damage code (e.g., "d6+1w")
    .dmg_good          string   — Good hit damage
    .dmg_amazing       string   — Amazing hit damage
    .range_str         string   — Range (e.g., "8/16/60" or "Personal")
    .damage_type       real     — DAMAGE_TYPE enum (LI/HI/EN)

  SAVE FORMAT: [name, skill_keyword, dmg_o, dmg_g, dmg_a, range, type_string]

ARMOR:  (PHB Table P41 p.189)
  .armor
    .name              string   — Armor name
    .li / .hi / .en    string   — Absorption values (e.g., "d6-1", "d4+1", "d4-1")

  SAVE FORMAT: [name, li, hi, en]

GEAR:
  .gear                array    — Array of gear name strings

FX (Unified modifier system):  (PHB Table P26 p.104, Table P27 p.108, Table P53 p.241)

    .name              string   — Must match global.fx_database entry
    .type              string   — "perk" / "flaw" / "cybertech"

    .active            bool     — Toggle for active/inactive

  Full FX data (keywords, modifiers, prereqs, size, cost) lives in

QUICK REFERENCE: Common operations for maintainers
================================================================================

TO ADD A NEW PERK/FLAW/CYBERTECH:
  1. Add entry to datafiles/fx_database.json
  2. No code changes needed. apply_fx_modifiers() handles it.

TO ADD A NEW WEAPON/ARMOR/GEAR:
  1. Add entry to datafiles/equipment.json
  2. No code changes needed. Equipment tab reads it automatically.

TO ADD A NEW SKILL:
  1. Add to datafiles/skills.json (under the appropriate broad skill)
  2. If it needs special keyword categories, update datafiles/keyword_tree.json

TO ADD A NEW SPECIES:
  1. Add to datafiles/species.json
  2. Update SPECIES enum in scr_chargen.gml (add before COUNT)

TO ADD A NEW TAB:
  1. Add draw_tab_xxx() in scr_draw_tabs.gml
  2. Add handle_tab_xxx() in scr_tab_handlers.gml
  3. Add dispatch in Draw_64 (line ~479) and Step_0 (line ~341)
  4. Increment _num_tabs and add tab name in Draw_64 (line ~288)

TO ADD A NEW CHANGELOG ENTRY:
  1. Add to top of entries[] in datafiles/changelog.json

TO MODIFY THE ROLL PIPELINE:
  All checks flow through meta_roll() (line 474 of scr_ui_helpers.gml).
  Hook into apply_fx_modifiers() for new modifier types.

CRITICAL INVARIANT:
  After ANY modification to hero data, call update_hero(hero).
  Forgetting this causes stale derived values (skill scores, durability, portrait).
